HTTP en un coup d'oeil

Beaucoup de développeurs Web utilisant HTTP ne connaissent pas vraiment le protocole en lui-même. Ce site fournit une explication de base du protocole HTTP 1.1 afin de faciliter le développement d'applications qui utilisent HTTP.

Cette "explication de HTTP" est un résumé incomplet, simplifié mais lisible de la spécification complète.
La spécification HTTP va bientôt être remplacée par plusieurs documents. Pour plus d'informations, vous pouvez visiter le wiki du groupe de travail HTTPbis.

Étant donné

Une URL, URI or IRI, que l'on appelle resource, avec le schéma http.

Exemple

http://example.com
http://www.google.com/?q=http
http://example.org/à-propos/mère

avec les capacités suivantes

  • GET

    Avec la méthode GET vous pouvez récupérer le contenu d'une resource.

  • PUT

    Avec la méthode PUT vous pouvez créer une nouvelle resource ou remplacer le contenu d'une resource existante.

  • POST

    Avec la méthode POST vous pouvez envoyer des données destinées à être traitées par la resource en question.

  • DELETE

    Avec la méthode DELETE vous pouvez supprimer une resource.

  • HEAD

    Avec la méthode HEAD vous pouvez vérifier l'état d'une ressource. Cette méthode est identique à la méthode GET, mais elle ne retourne aucun contenu.

  • OPTIONS

    Avec la méthode OPTIONS vous pouvez visualiser les fonctionnalités offertes par une ressource.

avec les réponses suivantes

  • 200 - Succès

    Une réponse dont le statut est compris entre 200 et 299 signifie que la requête a été réussie.

  • 300 - Redirection

    Une réponse dont le statut est compris entre 300 et 399 signifie que la requête doit être envoyée sur une autre resource.

  • 400 - Erreur client

    Une réponse dont le statut est compris entre 400 et 499 signifie que la requête envoyée par le client était malformée.

  • 500 - Erreur serveur

    Une réponse dont le statut est compris entre 500 et 599 signifie que la requête n'a pas pu être traitée à cause d'une erreur côté serveur.

Maintenant vous savez

Maintenant que vous en savez un peu plus sur HTTP 1.1, vous pouvez creuser un peu plus.

  • Commencez à développer de vraies architectures RESTful
  • Réfléchissez à votre usage des requêtes HTTP dans vos projets passés, et comment vous pourriez vous améliorer
  • Creusez un peu plus la spécification HTTP 1.1 et apprenez-en un peu plus à propos des entêtes ("headers") et tous les codes de retour ("status codes") et les méthodes que nous n'avons pas mentionnées ici
